### **[NY FUNKSJON] R1: AI Technical Debt Detector** 🟣

**Hva:** ML-modell som identifiserer teknisk gjeld som AI ofte lager.

**Vanlige AI debt patterns:**

| Pattern | Eksempel | AI-frekvens | Løsning |
|---------|----------|------------|---------|
| Hardcoded values | `MAX_ITEMS = 100` i 5 filer | 68% | Extract constant |
| Missing error handling | No try/catch, no null checks | 55% | Add defensive code |
| Unoptimized queries | No indexes, N+1 problems | 42% | Optimize queries |
| Inconsistent naming | `user` vs `usr` vs `u` | 75% | Enforce conventions |
| No logging/monitoring | Silent failures | 38% | Add observability |
| Missing input validation | Trust user input | 52% | Validate early |
| Memory leaks | Unreleased resources | 28% | Use cleanup patterns |
| Dead code | Functions never called | 45% | Remove unused code |

**Metodikk:**
- Scan for known AI antipatterns
- Use ML to find similar patterns
- Score likelihood of being AI-generated
- Suggest targeted fixes
- Track improvement over time

**For vibekodere:** Systemet vet hvilke feil AI-modeller gjør systematisk, og flagger dem automatisk.

**Impact:** +85% debt-detection-rate, -70% code-review-time for AI code

### **[NY FUNKSJON] R3: Vibekoding Debt Patterns** 🟣

**Hva:** Katalog av spesifikke problemer AI-generert kode har i vibekoding-kontekst.

**Vibekoding-spesifikke antipatterns:**

1. **Missing type safety**
   - AI generer JavaScript i stedet for TypeScript
   - No prop validation i React components
   - Fix: Add TypeScript, PropTypes, eller Zod validation

2. **Inefficient state management**
   - Re-renders whole tree instead of memoization
   - Missing React.memo() på components
   - Fix: Add useMemo, useCallback, React.memo

3. **N+1 database queries**
   - Loops doing database queries
   - No batch loading or joins
   - Fix: Use database joins or DataLoader pattern

4. **Missing async/await proper handling**
   - Unhandled promise rejections
   - Race conditions
   - Fix: Add error handlers, use Promise.all correctly

5. **Unoptimized API calls**
   - Missing request cancellation
   - No caching/memoization
   - Fix: Use React Query, SWR, or manual cache

6. **CSS-in-JS bloat**
   - Large CSS files for each component
   - No code splitting
   - Fix: Use CSS modules, Tailwind, or styled-components

7. **Missing accessibility**
   - No ARIA labels
   - Keyboard navigation broken
   - Fix: Add a11y testing, semantic HTML

8. **Unfinished error handling**
   - Generic "error" messages
   - No fallback UI
   - Fix: Add specific error types and boundaries

**Impact:** +90% vibekoding-debt-detection, enables focused improvements

### **[NY FUNKSJON] R4: Refactoring Impact Analyzer** 🟣

**Hva:** Forutsier konsekvenser av refactorings før de gjøres.

**Metodikk:**
- Analyser kodegrafer for dependencies
- Bruk historiske data for impact estimation
- Forutsig test-failure-sannsynlighet
- Estimér performance impact
- Forutsig breaking-change-risiko

**Eksempel:**
```
REFACTORING: Move 'calculatePrice()' from utils to services

IMPACT ANALYSIS:
================
Files affected: 12 imports
- src/api/checkout.ts (critical path)
- src/components/CartSummary.tsx (UI)
- tests/checkout.test.ts (need updates)

Estimated risk: MEDIUM
- Dependency change risk: 15% (breaking change)
- Test failure likelihood: 8% (2 tests may need updates)
- Performance impact: NEUTRAL (no change)

Recommended approach:
1. Keep old function as deprecated wrapper first
2. Update 3 critical imports
3. Update tests
4. Remove old function after 1 sprint

Timeline: 2-4 hours
Risk: LOW (with deprecation period)
```

**For vibekodere:** Før du refactorer, systemet sier deg nøyaktig hva som kan gå galt og hvordan du unngår det.

**Impact:** +70% refactoring-safety, -50% unexpected breakages

### Refactoring Techniques & Implementation
**Hva:** Guide implementation av refactorings på sikker, incremental måte.

- Test-driven refactoring (write test first)
- Pair programming for complex refactors
- Backwards compatibility where possible
- Feature flags for gradual rollout
- Monitoring for regressions

## GUARDRAILS

### ✅ ALLTID
- Write tests before refactoring (or add them first)
- Make small, incremental changes
- Run tests after each change
- Commit frequently (small commits)
- Get code review before merge
- Monitor after deployment
- Document decisions
- Educate team on patterns
- Measure impact after refactoring
- **[NY]** Use impact analyzer for safe changes
- **[NY]** Track AI antipatterns over time

### ❌ ALDRI
- Refactor without tests (you will break things)
- Refactor while adding features (pick one)
- Ignore performance implications
- Break backwards compatibility without migration plan
- Refactor just for "clean code" if business doesn't need it
- Skip the team buy-in (they need to maintain this)
- Optimize prematurely (measure first)
- Introduce complex patterns for "elegance"
- **[NY]** Ignore AI antipatterns (they compound over time)
- **[NY]** Refactor AI code without understanding debt origins

### ⏸️ SPØR
- Hvis kompleksitet high: Is it actually used this way? Can we simplify usage?
- Hvis coverage low: What's the business risk of this untested code?
- Hvis duplication exists: Is it really duplication or intentional separation?
- Hvis team hesitant: What's the risk they're worried about? How do we mitigate?
- Hvis performance slow: Is it actually slow or perceived slow?
- **[NY]** Hvis AI-generert kode: What specific antipatterns should we look for?
- **[NY]** Hvis massive debt: Should we rewrite or refactor incrementally?

### Funn 1: N+1 Query i getUsersWithPosts()
- **Alvorlighet:** Kritisk
- **Beskrivelse:** Database-query kjøres i loop, 100+ kall/dag. Typisk AI-antipattern.
- **Referanse:** AI Antipattern Catalog - N+1 Query Pattern

- **Alvorlighet:** Høy
- **Beskrivelse:** 8 steder mangler try/catch. Fører til silent failures i produksjon.
- **Referanse:** Clean Code - Error Handling
- **Anbefaling:** Legg til error boundaries og spesifikke error handlers

- **Alvorlighet:** Medium
- **Beskrivelse:** Mangler React.memo(), useMemo(), useCallback() på kritiske komponenter
- **Referanse:** React Performance Optimization
- **Anbefaling:** Legg til memoization på komponenter som re-rendrer ofte

### Funn 4: Hardcoded configuration values
- **Alvorlighet:** Medium
- **Beskrivelse:** Konfigurasjonsverdier hardkodet i 5 filer
- **Referanse:** 12-Factor App - Config
- **Anbefaling:** Flytt til environment variables
